## 👩‍💼 For the FinTech Product Manager

### What This App Does
A simulated iOS stock watchlist and portfolio tracker — **no real financial data, no real APIs, no PII**. All prices are generated by a random walk algorithm. Purpose: demonstrate production-grade iOS FinTech architecture patterns for learning and reference.

### Three Core User Journeys
1. **Investing Dashboard** — View simulated holdings, portfolio value, unrealised P&L
2. **Watchlist Dashboard** — Monitor stocks with price alerts (above/below threshold)
3. **Performance Dashboard** — 30-day price history charts with interactive crosshair

### Why It Matters for FinTech Product
| Product Concern | Architecture Response |
|---|---|
| **Regulatory: data exposure** | `MaskedValueView` hides all portfolio values in app switcher — PCI-DSS principle of least exposure |
| **Regulatory: access control** | Face ID / Touch ID biometric gate before any financial data is visible (LAContext) |
| **Regulatory: accessibility** | WCAG 2.1 AA compliance — VoiceOver, Dynamic Type, Reduce Motion on every financial tile |
| **Feature control without app store** | `FeatureFlagService` actor — remote JSON with safe OFF defaults. Kill any tile/dashboard remotely |
| **International markets** | Xcode 15 String Catalogs (`.xcstrings`) — full localisation architecture from day one |
| **User re-engagement** | `PortfolioWidget` (WidgetKit) — home screen widget with deep link back to app |
| **Push re-engagement** | `PriceAlert` model + `UNUserNotificationCenter` integration pathway documented |
| **Analytics** | `AnalyticsServiceProtocol` — swap any provider (Amplitude, Firebase, custom) without code change |

### Headless CMS — What It Means for PM
The v4.0 architecture introduces a **Headless CMS layer** (e.g. Contentful, Sanity, or internal JSON endpoint). This means:
- **New tile types, dashboard layouts, and promotional content** can be published without an App Store release
- **A/B test dashboard configurations** by serving different CMS content to user cohorts
- **Emergency content changes** (e.g. regulatory disclosure updates) deploy in minutes, not weeks
- **Non-technical teams** can manage tile copy, ordering, and visibility through a CMS dashboard

## 👔 For the Tech Executive

### Platform Choice — Why iOS 17+ Minimum
| Decision | Rationale |
|---|---|
| iOS 17+ minimum | Unlocks `@Observable`, SwiftData, Swift Charts, `#Preview` macro — all on day one. Zero legacy API debt from the first commit. |
| Swift 5.9 with `StrictConcurrency` | Compiler enforces thread-safety. Zero data races reach production. Swift 6 upgrade is a one-line change. |
| Zero external dependencies in `MobileDashboardKit` | No third-party supply chain risk in the reusable library. No `npm audit` equivalent surprises. |
| SwiftData (not Core Data) | Apple's modern persistence API — 60% less boilerplate, schema versioning built-in, migration plan defined. |

### Team Velocity — How MVVM-C Pays Off
- **Coordinators own navigation** — new screens are added without touching existing ViewModels. PR scope is minimal and predictable.
- **`AppEnvironment` DI container** — two feature teams can develop against mock dependencies in parallel. No environment collisions.
- **Headless CMS** — PM/design can iterate on tile layout without blocking engineering. Decoupled release cadence.
- **GitHub Actions pipeline** — lint → build → unit tests (≥80% coverage enforced) → UI tests → snapshot tests → TestFlight. Zero manual steps from merge to internal release.

### Risk Posture
| Risk | Mitigation |
|---|---|
| Data breach (device) | Keychain with `kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly` — financial config never in UserDefaults |
| Man-in-the-middle | `CertificatePinningDelegate` (OWASP M3) on all HTTPS — CMS, feature flag, analytics endpoints |
| Jailbroken device | `SecurityCheckService` — jailbreak + debugger detection logged to analytics |
| Visual regression | `swift-snapshot-testing` — tile rendering regressions are CI failures, not manual discoveries |
| Breaking changes in library | `MobileDashboardKit` has semantic versioning + typed `DashboardError` domain. Breaking changes are compiler errors in consumer apps. |

## 🏛️ For the Architect

### MVVM-C — What the Coordinator Pattern Solves

**The problem MVVM-C solves**: In standard MVVM, ViewModels accumulate navigation logic (`push`, `present`, `dismiss`). After 6 months, ViewModels become massive and screens are tightly coupled — you cannot reuse `StockDetailView` from both `WatchlistCoordinator` and `InvestingCoordinator` without duplication.

**The MVVM-C solution**: Navigation is extracted into a dedicated `Coordinator` layer. ViewModels emit **intents** (e.g. `.showStockDetail(ticker:)`). The Coordinator owns the `NavigationPath` and decides *how* to navigate.

```
User Tap → View → ViewModel (intent) → Coordinator → NavigationPath update → SwiftUI renders
                  ↑                      ↓
               Pure domain          No SwiftUI import
               logic only           in ViewModel
```

**Coordinator tree for StockWatchlist:**
```
AppCoordinator (root — biometric gate, tab selection)
├── InvestingCoordinator  (NavigationPath for Investing tab)
│   ├── HoldingDetailCoordinator
│   └── AddHoldingCoordinator
├── WatchlistCoordinator  (NavigationPath for Watchlist tab)
│   ├── StockDetailCoordinator
│   └── AlertDetailCoordinator
├── PerformanceCoordinator
└── SettingsCoordinator
```

### Clean Architecture Layer Dependencies
```
┌─────────────────────────────────────────────┐
│  Presentation   SwiftUI Views + ViewModels   │
│  (can import Domain, cannot import Data)     │
├─────────────────────────────────────────────┤
│  Coordinators   Navigation + Deep Links      │
│  (can import Presentation + Domain)          │
├─────────────────────────────────────────────┤
│  Domain         Pure Swift — NO frameworks   │
│  Models + Protocols + Use Cases              │
├─────────────────────────────────────────────┤
│  Data           SwiftData + PriceFeedActor   │
│  (implements Domain protocols)               │
├─────────────────────────────────────────────┤
│  CMS            Headless content layer       │
│  (TileConfig + DashboardConfig from JSON)    │
└─────────────────────────────────────────────┘
         Dependencies flow DOWNWARD ONLY
```

### Key Design Decisions
| Decision | Pattern | Rationale |
|---|---|---|
| Tile rendering | `TileRenderable` protocol + `@ViewBuilder` | No `AnyView` type erasure — SwiftUI diff engine remains fully effective |
| Reactive state | `@Observable` macro | Replaces `ObservableObject/@Published` — compiler-proven observation graph, iOS 17+ |
| Concurrency | `actor PriceFeedActor` + `AsyncStream` | Data races impossible at type-system level. UUID-tagged continuations. |
| Navigation | `Coordinator` protocol + `NavigationPath` | Screens are 100% reusable across flows. Deep links handled at AppCoordinator level. |
| CMS rendering | `CMSTileFactory` + `TileKind` enum | Compile-time exhaustive tile catalogue. New tile = add enum case + factory case. |
| Persistence | SwiftData `@Model` + `VersionedSchema` | Schema migration is declarative. Data loss on upgrade is impossible. |
| DI | `AppEnvironment` struct | No 3rd-party DI framework. `live()` and `mock()` — injectable everywhere including SwiftUI `#Preview`. |

### Critical Swift Rules in This Codebase
```swift
// ✅ Monetary values — ALWAYS Decimal, never Double/Float
var price: Decimal = 182.50

// ✅ @Observable macro — never ObservableObject on iOS 17+
@Observable @MainActor final class InvestingViewModel { }

// ✅ actor for concurrent state — never class + DispatchQueue.main
actor PriceFeedActor { }

// ✅ Protocol DI — never static let shared singleton
let env = AppEnvironment.live()    // production
let env = AppEnvironment.mock()    // tests and #Preview

// ✅ @ViewBuilder protocol — never AnyView type erasure
protocol TileRenderable {
    associatedtype Body: View
    @ViewBuilder @MainActor func body() -> Body
}

// ✅ Coordinator for navigation — never push/present in ViewModel
coordinator.handle(.showStockDetail(ticker: "AAPL"))

// ✅ CryptoKit.SHA256 — never CC_SHA256 (CommonCrypto)
let fingerprint = SHA256.hash(data: certData)
    .map { String(format: "%02x", $0) }.joined()
```

### TDD Red-Green-Refactor Cycle (v4.0)
Every feature follows this exact sequence — no exceptions:
```
1. RED    Write a failing XCTestCase that describes the BEHAVIOUR
2. GREEN  Write the minimum code to make the test pass
3. REFACTOR Clean up — the test still passes after every change
```

Failure analysis is a first-class workflow: when a test fails, the `FailureAnalysisService` classifies the failure as `domainLogic`, `concurrency`, `uiRendering`, `persistence`, or `network` and routes to the correct remediation path.

## Design Rules — Quick Reference Card

| Rule | Detail |
|---|---|
| Money | `Decimal` only — never `Double`/`Float` in domain or UI |
| State | `@Observable` macro — never `ObservableObject`/`@Published` |
| Concurrency | `actor` types — never `class` + `DispatchQueue.main.async` |
| Navigation | Coordinator intents — never push/present in ViewModel |
| Services | `AppEnvironment` DI — never `static let shared` singleton |
| Sensitive UI | `MaskedValueView` — all monetary values when `scenePhase != .active` |
| Auth | Biometric gate before portfolio data — always |
| Secrets | Keychain — never `UserDefaults` for anything financial |
| Errors | `Result<T, E>` or `async throws` — never silent swallowing |
| Strings | `String(localized:)` — no hardcoded English literals |
| Tests | Red → Green → Refactor — no feature merged without a test |
| Coverage | ≥80% enforced in CI — PRs fail below this threshold |
| Style | `swiftlint --strict` in CI — zero violations on `main` |
